AmigaOS

Operating System used by the Amiga range of home computers

Kickstart and Workbench versions

Kickstart - The kernel. In ROM, except for the A1000
Workbench - The user interface, invoked with the command "loadwb", although a good chunk of it loads anyway to give you the CLI. On disk.
Any workbench version can be used as long as it's major release (number before the decimal point) matches that of the Kickstart
Noteable releases:
  • Kickstart 1.2 - Brilliant white opening screen showing a hand holding a disk. Catchy. Little hard drive support, but it was there. Workbench was a deliberate high-constrast thing of blue and orange. Quite painful to use, and no support to create a new directory. You had to duplicate an existing one. Also, deleted items *had* to be dragged to a trashcan. If there wasn't a trashcan on your disk, you couldn't delete. No commands in memory, not even CD. Two floppy drives or a hard drive essential if you wanted to do anything remotely serious
  • Kickstart 1.3 - Almost identical, but it supported bootable hard drives.
  • Kickstart 2.04 - Low contrast pleasant black/white/grey appearance. Boot screen changed to a purple animation of a disk being inserted into a drive. Introduced delete and new folder commands. Made a lot of commands resident, such as cd, so no need to keep reinserting the Workbench disk. Strangely, commands such as DIR were still disk-based. What most non-Amiga users will be familiar with, due to the popularity of the A500+ and the A600
  • Kickstart 3.01 - Very similar in appearance to 2.04, but the changes under the bonnet were quite significant. Firstly was the support for cd drives, even bootable ones with 3.01 (as opposed to 3.0). Support for graphics cards like the Picasso (see later). Secondly was the introduction of "datatypes". This is essentially identical to the "class-type" idea on PCs, whereby a file _type_ could be assosciated with a program. Small moves... Moved a lot of boot-files to the ROM to increase speed

    Guru Meditation

    If a software failure caused the Amiga to reboot, a flashing red box appeared with the phrase "Guru Meditation". This is supposed to be relaxing. The following codes described the error code and location in memory where it happened. Not much consolation to know that error 0x00000003 means divide by zero when you've just lost 12 hours work. Replaced by "Software Failure" with Workbench 2.04

    Methodology

    Devices and APIs were handled in a very dynamic way by the Amiga. Upon bootup, a variety of directories would be checked:
The C directory, contained commands which could be called from the command line. The default path would point here
The DEVS directory contained drivers for devices (not hard or floppy drives) which might be connected. They had catchy names such as cd.device or scsi.device
The L directory contained disk-validator, a file used to check integrity of the disk. Luckily this silly idea was abolished with Kickstart 2.04
The LIBS directory contained the API for the OS. Some were fairly sensible, such as ARP.library (later ASL.library) (the GUI), and powerpacker.library (for on the fly decompression), to the bizarre mathieeeedoubbas.library and mathieeeedoubbtrans.library (maths functions, I presume)
The S directory contained the catchily-named startup.sequence which was a script called upon bootup
The T directory was always a mystery to me since I never found anything in it
The Amiga never used extensions to identify filetypes. Instead, it relied on the application checking the first few bytes of the file before using it. Hey, it worked. Once datatypes were introduced, the OS could identify and assosciate files on the same principle
This does mean that executable files had to be tagged as such (with the X flag) otherwise the OS would not know they weren't data files
Another flag was the S tag, which meant the file contained a list of commands (like a batch file) to run. If you didn't set the flag, you could still treat a file as such with the EXECUTE command
Native support for PC disks (double density only)
The Amiga uses the same CR/LF format as Unix, so there is no problem reading ASCII files. A tool called Crossdos could be invoked to run in the background (like a daemon or TSR), which would silently convert the end of line characters for you. Wasn't that nice? It also uses Unix backslashes, and many of the commands have similar attributes and names. Ports of RexX and eMacs were available (aRexX and MicroEmacs respectively)
